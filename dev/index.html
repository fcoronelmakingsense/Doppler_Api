Getting started

Since the API is based on REST principles, it's very easy to write and test applications. You can use your browser to access URLs, and you can use pretty much any HTTP client in any programming language to interact with the API.

Getting an API Key

Most of the Doppler API services require to call them on a behalf of a user, so it is required an API Key to be included in each request.

See Where do I find my API Key?.

Using API Key §

The API Key allows you to make requests to the API on a behalf of a user.

GET https://restapi.fromdoppler.com?access_token=...

You can pass the API Key in the query params like shown above, but a cleaner (and safer) approach is to include it in the Authorization header.

GET https://restapi.fromdoppler.com

Authorization: token OAUTH-TOKEN

Dealing with errors §

Suppose that you have reached the max list number allowed for your account and you try to create a new list. Doppler API will return you something like this:

{
  "title": "Maximum number of lists reached",
  "status": 400,
  "errorCode": 3,
  "detail": "It reached the maximum number of Lists. Can not create new ones.",
  "type": "https://restapi.fromdoppler.com/docs/errors/400.3-maximum-number-of-lists-reached"
}

Notice about type field, it refers to a page where you can get more information about the error and how to solve it. See the section Errors for the complete list of possible problem types.

Dealing with dates and time §

DateTime §

The follow is a subscribers list representation, from a Doppler API response:

{
  "listId": 509702,
  "name": "My First List",
  "currentStatus": "ready",
  "subscribersCount": 0,
  "creationDate": "2015-09-01T15:16:32.75+00:00",
  ...

As you can see creationDate represents a DateTime value. To avoid ambiguities, Doppler exposes this kind of data using Coordinated Universal Time (UTC) and making offset explicit (+00:00).

Examples of accepted DateTime values are: 2015-08-15T09:00+04:00, 2015-05-21T09:00:36-03:00, 2010-01-29T09:00:36.515+00:00 and 2015-01-29T09:00:36Z.

DateTime filtering §

Requesting to some collections, for example Unsubscribed Subscribers, allows you to filter by a range of time giving the parameters from and to following the follow rules:

  • DateTime values should be explicit about timezone offset.

    • Valid: ...?from=2015-05-21T09:00:36-03:00&to=2015-01-29T09:00:36Z.
    • Invalid: ...?from=2015-05-21T09:00:36&to=2015-01-29T09:00:36.

  • If parameter is missing, the filter is not applied.

    • ...?from=2015-05-21T09:00:36-03:00&to=2015-01-29T09:00:36Z means between 2015-05-21T09:00:36-03:00 and 2015-01-29T09:00:36Z.
    • ...?from=2015-05-21T09:00:36-03:00 means after 2015-05-21T09:00:36-03:00.
    • ...?to=2015-01-29T09:00:36Z means before 2015-01-29T09:00:36Z.

  • from filter is not strict (inclusive)

    • ...?from=2010-01-29T09:00:36.515+00:00 includes events occurred exactly at that time and after.

  • to filter is strict (exclusive)

    • ...?to=2010-01-29T09:00:36.515+00:00 does not include events occurred exactly at that time, only events occurred before.

Important: Take into account that parameters in query string should be encoded, so + character should be replaced by %2B. For example, if you want to get all unsubscriptions done after January 1, 2015 at 3 PM in Rome, Italy (UTC +2 hours), URL will be: https://restapi.fromdoppler.com/accounts/test@test.com/unsubscribed?from=2015-01-01T03:00%2B02:00.

Date only values

Date only values are simpler. Doppler API accepts these formats: 2015-08-20 and 20150820.

Asynchronous Tasks §

There are some task that are not executed instantly, for example Import Subscribers.

If callback property is set, when import process finish, Doppler API will execute a POST request against that URL, in our example: http://my.own.server.com/my/own/resource.

Request:

POST https://restapi.fromdoppler.com/accounts/test@test.com/lists/509719/subscribers/import

Authorization: token 22495B800297B754C6EF66FA2C5C1902
Content-Type: application/json

{
  "fields": [],
  "items": [ 
    { "email": "test@test.com" }
  ],
  "callback": "http://my.own.server.com/my/own/resource"
}

Subscriber Field Values §

Subscribers have a set of string key-value pairs called fields (see subscriber.json and field-value.json), to get the full list of account's field definitions use Fields collection resource.

Field value is always a string value, but depending on field type it has a specific format:

  • boolean: true or false (case sensitive).
  • number: A series of digits with or without dot followed by another series of digits.
  • string: Any text value (400 character max).
  • date: It accepts any valid date or date-time (see Date and time section), but only stores date part. Recommended format is yyyy-MM-dd.
  • gender: M for male and F for female (case sensitive), anything else for not specified.
  • country: ISO 3166-1 alpha-2 country code (case insensitive).

Special behavior during subscribers import: If a value cannot be parsed following the previous rules, the previous field value will not be changed without any warning.
